# Plano: Transição de Tesseract para EasyOCR

## Objetivo

Substituir o Tesseract OCR (`pytesseract`) pelo **EasyOCR** (deep learning) no microserviço Python de OCR (`ocr-service/`), mantendo a mesma interface API e estrutura de dados de saída.

---

## Escopo

A transição abrange **apenas o microserviço Python** (`ocr-service/`). O PHP mantém chamadas ao mesmo endpoint FastAPI, sem alterações no lado Laravel.

**Nota:** Existem duas referências Tesseract diretas no PHP (`runTesseractOnImage` em `PurchaseOrderController.php:1495` e `GoodsReceiptController.php:996` que usam `thiagoalessio/tesseract_ocr`). Estes são caminhos de fallback separados e **não** são alterados neste plano.

## 2. `ocr_processor.py` — Núcleo da Transição

### 2.1 EasyOCR Reader Singleton

EasyOCR `Reader` carrega modelos pesados (~100-200MB). Deve ser inicializado **uma única vez** e reutilizado:

```python
import easyocr

_reader: easyocr.Reader | None = None

def get_reader() -> easyocr.Reader:
    global _reader
    if _reader is None:
        _reader = easyocr.Reader(['pt', 'en'], gpu=True)  # gpu=False se sem GPU
    return _reader
```

### 2.2 Funções a Substituir / Reescrever

#### `extract_text_with_layout()`

**Antes:** Tenta PSM 4, 3, 11 com Tesseract e escolhe o melhor.

**Depois:** Usa `reader.readtext()` uma única vez:

```python
reader = get_reader()
results = reader.readtext(processed_rgb)  # EasyOCR espera RGB
# results: [(bbox, text, confidence), ...]
# bbox: [[x1,y1], [x2,y2], [x3,y3], [x4,y4]]
```

**Mapeamento de dados:**

| Campo Tesseract (atual) | Fonte EasyOCR |
|------------------------|---------------|
| `raw_text` | Join de todos os `text` ordenados por posição |
| `lines` | Grouping vertical (Y-overlap) + sorting horizontal |
| `tables` | `detect_tables(lines)` — **sem alterações** |
| `hocr` | Não disponível → string vazia `""` |
| `psm_used` | Não aplicável → `None` ou `'easyocr'` |

#### `_assemble_lines_from_ocr_data()`

**Substituir completamente** por lógica nova:

1. Ordenar detecções por coordenada Y (topo do bbox)
2. Agrupar detecções cujos intervalos verticais se sobreponham (mesma "linha")
3. Ordenar cada grupo por coordenada X (esquerda para direita)
4. Juntar texto do grupo com espaços → 1 linha
5. Calcular `bbox` consolidado da linha (min X, min Y, max W, max H)
6. `confidence` = média das confianças dos deteções na linha

#### `extract_text_simple()`

Substituir `pytesseract.image_to_string()` por `reader.readtext()` e concatenar textos.

### 2.3 Pré-processamento de Imagem

**Atual:** `preprocess_image()` faz:
1. Load image (BGR)
2. Perspective correction
3. Upscale ≥2000px
4. Sharpen
5. Grayscale
6. Deskew

**Proposto:**
- **Manter** perspective correction e deskew — continuam benéficos
- **Remover** sharpen e upscale agressivo — EasyOCR lida bem com várias resoluções; modelos pré-treinados esperam resoluções próximas do treino (tipicamente ~300-600px de altura)
- **Alterar** saída para **RGB** (EasyOCR espera RGB; atualmente retorna grayscale)
- Adicionar parâmetro opcional para EasyOCR: `paragraph=True` se quiser parágrafos, `min_size`, `width_ths`, `height_ths` para controlar grouping

**Resolução alvo:** Manter upscale mínimo (ex: 800px no lado mais curto) para evitar degradação em imagens muito pequenas.

### 2.4 Conversão bbox

EasyOCR devolve `[[x1,y1],[x2,y2],[x3,y3],[x4,y4]]` (formato polígono).
Converter para `[x, y, w, h]` (formato retângulo atual):

```python
x = min(x1, x2, x3, x4)
y = min(y1, y2, y3, y4)
w = max(x1, x2, x3, x4) - x
h = max(y1, y2, y3, y4) - y
```

### 2.5 Remover Código Obsoleto

- `order_points()` — usado por perspective correction, **manter**
- `_try_perspective_correction()` — **manter**
- `correct_perspective()` — **manter**
- `apply_clahe()` — atualmente não chamado no pipeline, pode remover ou manter
- `sharpen_image()` — pode remover (não necessário para EasyOCR)
- `auto_threshold()` — pode remover (não necessário para EasyOCR; EasyOCR usa grayscale/RBG diretamente)

## 6. Potenciais Riscos e Trade-offs

| Aspecto | Tesseract | EasyOCR |
|---------|-----------|---------|
| **Instalação** | Rápida (~50MB) | Pesada (~800MB-2GB por torch) |
| **Velocidade (CPU)** | Rápido (1-3s) | Mais lento (3-10s sem GPU) |
| **Precisão** | Bom em docs estruturados | Melhor em docs variados/menor qualidade |
| **Português** | Suportado (`por`) | Suportado (`pt`) |
| **HOCR** | Sim | Não |
| **PSM modes** | Sim (multi-modo) | Não (single-pass) |
| **GPU** | Não (usa CPU) | Sim (CUDA) |
| **Pré-processamento** | Crítico (binarização, threshold) | Menos crítico |

## 8. Questões em Aberto para Decisão

1. **GPU vs CPU:** Usar `gpu=True` (se CUDA disponível) ou `gpu=False` (CPU-only)?
2. **Linha de agrupamento:** O atual `_assemble_lines_from_ocr_data` agrupa palavras por `block_num`/`line_num` do Tesseract. Com EasyOCR, precisamos de heurística Y-overlap. Que tolerância vertical para considerar mesma "linha"?
3. **Manter upscale?** EasyOCR funciona nativamente a resoluções baixas. Podemos reduzir/remover o upscale para maior velocidade.
4. **Compatibilidade do campo `tesseract_available`:** Renomear ou manter no `/health`?
