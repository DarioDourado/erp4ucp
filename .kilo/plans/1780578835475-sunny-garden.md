# Plan: Fix OCR Premature DB Creation + Parsing Errors

## Diagnóstico

### Problema 1: Produtos/fornecedores criados antes da validação do utilizador

`enrichPurchaseOrderDataWithDatabase()` (linha 1956) chama `findOrCreateSupplierFromOcr()` e `findOrCreateProductFromOcr()`, que **criam registos na BD** durante a fase de preview OCR. Isto é chamado em todos os caminhos (Python microservice, PHP fallback, Tesseract). O resultado: produtos `ART-4`, `ART-5`, etc. são criados antes de o utilizador clicar "Criar Encomenda".

**Solução:** Separar "find" de "create". O `enrich` deve apenas **resolver correspondências** com produtos/fornecedores existentes. A **criação** só deve acontecer no `PurchaseOrderStore`.

### Problema 2: Parsing incorreto do documento NE-2026-09002

Ao processar o PDF/HTML, o sistema extraiu linhas incorretas. Análise linha a linha:

| Resultado OCR | Esperado | Causa Raiz |
|---|---|---|
| ART-4: "ÂNTICO CASH & CARRY, LDA Cliente: Rua..." qty=0 | **Não é produto** (é o cliente) | `isLineExcluded` não tem `cliente` nos padrões; a linha contém números (NIF 502345671) que o parser confunde com qtd/preço |
| ART-5: "Cerveja Super Bock..." qty=40 price=14.40 | ✅ Correto | — |
| ART-6: "Vinho Verde..." qty=**239** price=**192.00** | qty=60 price=3.20 | **LLM/Python alucinou** estes valores (239×192=45888). O PHP fallback nunca corre porque `$hasPrices=true` (192>0). O LLM devolveu dados errados e o sistema confiou. |
| ART-10: "sl IVA" qty=948.2 | **Não é produto** (Subtotal) | OCR leu "Subtotal (s/ IVA)" como "sl IVA"; `isLineExcluded` tem `subtotal` mas "sl" não faz match |
| IVA: qty=196.17 | **Não é produto** (linha de IVA) | Linha de imposto passa como produto |
| ART-11: "ções: Entrega..." qty=253 price=222 | **Não é produto** (Observações) | OCR leu "Observações" como "ções"; `isLineExcluded` tem `observaç` mas "ções" (sufixo) não faz match. Tel "253 222 111" é interpretado como qty=253 price=222 |

**Causas raiz do parsing:**

1. **`isLineExcluded` incompleto** — faltam `cliente`, padrões para moradas com NIF embutido, e correspondência parcial de palavras (ex: "ções" vs "observações")
2. **LLM produz dados errados e o PHP fallback nunca é acionado** — o checker (linha 782) só dispara se `$hasPrices=false` ou `$hasDescriptions=false`. Se o LLM devolve preços errados (≠0), o fallback nunca corre.
3. **Sem validação semântica das descrições** — moradas, totais, IVA e observações passam como "produtos" porque ninguém verifica se a descrição parece um produto real.

## Plano de Implementação

### Parte A: Desacoplar "find" de "create"

**Ficheiro:** `app/Http/Controllers/Actl/PurchaseOrderController.php`

#### A1. Adicionar parâmetro `$createIfMissing` a `findOrCreateSupplierFromOcr`
- Assinatura: `findOrCreateSupplierFromOcr(array $parsedSupplier, bool $createIfMissing = false): array`
- Quando `false`, apenas pesquisa; não cria. Retorna `['found' => false, 'model' => null]` se não existir.
- Passar `true` apenas quando chamado do `PurchaseOrderStore`.

#### A2. Adicionar parâmetro `$createIfMissing` a `findOrCreateProductFromOcr`
- Mesmo padrão: `findOrCreateProductFromOcr(array $parsedLine, bool $createIfMissing = false): array`
- Quando `false`, não cria produtos nem famílias nem unidades.

#### A3. Passar `$createIfMissing = false` do `enrichPurchaseOrderDataWithDatabase`
- Preservar os dados OCR (código, descrição, preço) mesmo quando produto não encontrado.
- Adicionar campo `ocrRaw` com os dados originais para uso posterior na criação.

#### A4. Criar/atualizar `PurchaseOrderStore` para resolver produtos no momento da criação
- Se existir `ocr_purchase_order_data` na sessão, chamar `findOrCreateProductFromOcr` e `findOrCreateSupplierFromOcr` com `$createIfMissing = true`.
- Caso contrário, usar o fluxo normal do formulário.

### Parte B: Corrigir o parsing PHP (fallback e validação)

#### B1. Expandir `isLineExcluded` (linha 1885)
Adicionar aos padrões de exclusão:
- `cliente` — evita que moradas de cliente sejam tratadas como produto
- Deteção de NIF embutido na linha: se a linha contém um número de 9 dígitos precedido de "NIF" ou no formato `\d{9}`, excluir
- Padrão para fragmentos de palavras-chave: `(?:sl|subtotal|sub\s*total)` e `(?:obs|observa|ções)` como padrões no conteúdo da linha
- Linhas que começam com número grande (>100) seguido de vírgula/ponto e depois termo fiscal (IVA, Total, etc.)

#### B2. Melhorar o fallback checker (linha 782)
- Sempre que o LLM devolver dados, **comparar** com o resultado do parser PHP.
- Se o parser PHP encontrar MAIS linhas que o LLM → usar PHP.
- Se o parser PHP encontrar preços significativamente diferentes (>50% discrepância) → usar PHP.
- Adicionar sanity check: se uma linha tem unitPrice > 100 e quantity > 100, provavelmente é linha de total/agregado.

#### B3. Adicionar filtro pós-processamento em `extractPOTabularLines`
- Após extrair todas as linhas, filtrar as que parecem linhas de resumo:
  - Descrição é apenas números + caracteres especiais
  - Descrição contém fragmentos como "iva", "total", "subtotal", "obs"
  - A descrição tem < 3 caracteres alfabéticos significativos
- Filtrar linhas cuja descrição contém padrões de morada (Rua, Av., NIF, etc.)

#### B4. Melhorar heurística de 3 números no Passo 3 (linha 1486)
- Adicionar verificação: se `price > quantity * 10`, provavelmente o preço foi confundido com total → tentar permutação alternativa dos 3 números.
- Ex: para `[239, 192, 45888]`, testar `[60, 3.20, 192]` — mas isso requer que o OCR tenha extraído os números corretos. O problema real está no texto OCR de entrada.

---

### Parte C: Validação semântica no enriquecimento

#### C1. Adicionar validação de descrição em `enrichPurchaseOrderDataWithDatabase`
- Rejeitar linhas cuja descrição contenha padrões de:
  - Morada ("Rua", "Av.", "Lote", "Nº", "\d{4}-\d{3}")
  - NIF (9 dígitos consecutivos)
  - Termos fiscais ("IVA", "Total", "Subtotal", "Emissão", "Documento")
  - Telefones (9+ dígitos com espaços)
- Logar como warning e pular a linha.

### Parte D: Validação cruzada de sub-totais

#### D1. Extrair sub-totais do documento (backend)
- Em `parsePurchaseOrderDocument`:
  - Adicionar regex para extrair subtotal: `/Subtotal\s*(?:\(?s\/?\s*IVA\)?)?\s*:?\s*([\d.,]+)\s*€?/i`
  - Adicionar regex para extrair total: `/(?:TOTAL|Total\s+Geral)\s*:?\s*([\d.,]+)\s*€?/i`
  - Adicionar regex para linhas de IVA: `/(?:IVA|VAT)\s*(?:\(?\d+%?\)?)?\s*:?\s*([\d.,]+)\s*€?/i`
  - Devolver `documentSubtotal`, `documentTotal`, `documentTaxes[]` no array de saída
- Em `convertLLMFormatToInternal`: propagar os totais do LLM se existirem
- Em `enrichPurchaseOrderDataWithDatabase`:
  - Calcular `calculatedSubtotal = sum(lines.quantity * lines.unitPrice)` (apenas linhas enabled)
  - Comparar com `extractedSubtotal`
  - Calcular `discrepancy = abs(calculatedSubtotal - extractedSubtotal)` e `discrepancyPercent`
  - Adicionar campo `validation` ao output:
    ```php
    'validation' => [
        'subtotal' => ['extracted' => float, 'calculated' => float, 'discrepancy' => float, 'discrepancyPercent' => float, 'status' => 'ok'|'warning'|'error'],
        'total' => ['extracted' => float, 'calculated' => float, ...],
    ]
    ```
  - Thresholds: < 2% → ok (verde), 2-10% → warning (amarelo), > 10% → error (vermelho)

#### D2. Exibir validação no frontend (purchaseOrder_ocr.blade.php)
- Na secção "2. Dados Extraídos", após a tabela de linhas, adicionar um bloco de validação:
  - Card com "✅ Validação de Totais" ou "⚠️ Discrepância detetada"
  - Tabela com: Subtotal Documento, Subtotal Calculado, Diferença, %
  - Se houver discrepância > 10%, mostrar alerta a recomendar edição manual
  - Se houver totais de IVA extraídos, mostrar também a validação cruzada:
    - IVA calculado = sum(lines com taxRate X) vs IVA extraído do documento

### Parte E: Seleção de linhas pelo utilizador

#### E1. Adicionar campo `enabled` a cada linha na sessão
- Em `enrichPurchaseOrderDataWithDatabase`: adicionar `'enabled' => true` a cada linha.
- Em `updatePurchaseOrderOCRData`: suportar o campo `enabled` nas linhas enviadas pelo frontend.
- Modificar o método para também aceitar `lines.*.enabled` (booleano).

#### E2. Novo endpoint para toggle rápido de linhas (opcional, para performance)
- Rota `POST purchaseOrder.toggleOcrLine` → controller method `toggleOcrLine`
- Aceita `{ lineIndex: int, enabled: bool }`
- Atualiza diretamente `session('ocr_purchase_order_data.lines.N.enabled')`
- Alternativa: usar o `updatePurchaseOrderOCRData` existente (mais simples, menos código)

#### E3. Modificar `PurchaseOrderAdd` para filtrar linhas desativadas
- Em `PurchaseOrderAdd` (linha 68-102):
  - Ao ler `session('ocr_purchase_order_data')`, filtrar linhas com `enabled === false`
  - Apenas linhas com `enabled !== false` (ou `enabled === true`) são passadas como `initialLines`
  - Se não houver linha enabled → mostrar aviso no formulário

#### E4. Frontend — checkboxes na tabela de linhas (purchaseOrder_ocr.blade.php)
- Na vista de leitura (`ocr-read-view`):
  - Adicionar coluna de checkbox `<th style="width:5%;"><input type="checkbox" id="select-all-lines" checked></th>`
  - Cada linha ganha `<td><input type="checkbox" class="line-selector" data-index="N" checked></td>`
  - Estado inicial: todas checked (enabled=true)
- Na vista de edição (`ocr-edit-view`):
  - Adicionar coluna de checkbox também
  - Sincronizar estado com a vista de leitura
- JavaScript:
  - `#select-all-lines` → toggle todas as `.line-selector`
  - `.line-selector` change → enviar AJAX para `purchaseOrder.updateOCRData` com o estado enabled de todas as linhas
  - Mostrar contador de linhas selecionadas (ex: "5 de 9 linhas selecionadas")
  - Atualizar o sumário da secção 3 com o número de linhas selecionadas
- Secção "3. Criar Encomenda":
  - Mostrar "X linhas selecionadas de Y extraídas"
  - Se 0 linhas selecionadas → desativar botão "Criar Encomenda"

#### E5. Comportamento do botão "Guardar" na edição
- Quando o utilizador edita e guarda (`saveEditedData`), incluir o estado `enabled` de cada linha no payload
- O `updatePurchaseOrderOCRData` atualiza o campo `enabled` na sessão
